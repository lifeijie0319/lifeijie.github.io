防止短信轰炸的解决方案

发表于 2021-12-04 | 分类于 software_enginering ， practice

背景描述

有一个短信系统，功能很简单，负责发送短信验证码和验证短信验证码。现在需要出一个方案，预防可能的短信轰炸问题。假设要求一分钟内同一个手机号不能接受3次以上的短信发送请求。
系统是tornado多进程处理框架，使用SQLAlchemy连接mysql数据库。

方案一

描述：来一笔短信发送请求，查询一分钟内登记的短信发送明细记录，如果小于3笔，本次可以发送，否则不能发送。
分析：最简单的思路，但是有并发问题，导致实际发送的数量多于3笔。假设数据库中本没有某手机号的记录，此时同时接受100笔请求，100笔请求同时查数据库，发现1分钟内短信发送条数都是0，导致最后发送了100笔短信验证码，不符合要求。

方案二

描述：使用乐观锁。另加一张表A，以手机号为唯一主键phone，另有字段次数count，一分钟起始时间start_time和最后更新时间last_modified。每次请求过来入库，如果违反唯一约束，说明已有记录，捕捉异常忽略。查看当前时间now，如果now-start_time < 1分钟，查看count，如果count < 3，使用乐观锁更新count=count+1，失败重试；如果count >= 3，拒绝此交易。如果now-start_time >= 1分钟，使用乐观锁更新当前记录的start_time和last_modified为now，成功发送记录，失败重试。
分析：能够解决并发的问题。但是一是逻辑较为复杂，存在多次重试与失败的可能；二是其实这是把时间分段，即以第一次入库或者更新start_time为起点的一分钟，并不是任意的一分钟。比如14:15:00发送第一笔请求，14:15:55发送第二笔，14:15:56发送第三笔，14:16:02发送第四笔，14:16:03发送第五笔，站在14:16:00这个时间点往前往后各取30s，这一分钟内发送了4笔。

方案三

描述：使用悲观锁。另加一张表A，以手机号为唯一主键phone。每次请求过来入库，如果违反唯一约束，说明已有记录，捕捉异常忽略。入库完成之后，使用select…for update语句对手机号对应记录加排他锁，其它同样手机号的请求处理到这里需排队等候。加锁之后在同一事务内查看一分钟内的明细记录条目（明细表B），如果超过3就拒绝发送，否则新增明细入库，提交事务，然后发送短信验证码。
分析：悲观锁效率较乐观锁低，但是因为是对手机号记录加行锁，不对其它手机号记录造成影响，正常情况下并发不高，而且逻辑比乐观锁简单清晰，因而目前来看这是更加合适的解决方案。
