Lock

发表于 2022-03-13 | 分类于 language ， java ， basic ， concurrent

Lock接口

Lock接口用于实现与synchronized的类似的多线程并发访问控制功能。

主要方法

  • void lock(); 线程尝试获取锁，获取不到就在同步队列里一直重试，知道获取到为止。
  • void lockInterruptibly() throws InterruptedException; 类似lock()方法，但是可以被中断。
  • boolean tryLock(); lock()方法的非阻塞版本，获取不到锁时立刻返回false。
  • boolean tryLock(long time, TimeUnit unit) throws InterruptedException; 在time时间内一直重试获取锁；成功返回true；可被中断，中断抛出异常；超时未获取到返回false。
  • void unlock(); 锁的持有者释放锁
  • Condition newCondition(); 创建一个与该锁绑定的条件变量。Condition实例调用await()方法会先释放锁然后进入等待队列，而await()直到重新获取到锁（一般是由于通知）才会返回。

ReentrantLock

ReentrantLock是Lock接口的一个实现类。

重入锁

当线程获得对象的锁之后，再次请求该对象的锁可以成功，称该类锁为重入锁。synchronized和ReentrantLock都是重入锁。

公平锁

CPU在调度线程的时候是在等待队列里随机挑选一个线程，不保证线程对锁的获取是先到先得，这就是非公平锁。非公平锁会有饥饿现象，即有的线程一直获取不到锁。保证线程FIFO地获取锁，这就是公平锁，但是公平锁有维护有序队列的额外开销。synchronized是非公平锁，ReentrantLock可以通过构造器参数控制是否公平，默认是非公平锁。

ReentrantLock与synchronized地区别

角度 synchronized ReentrantLock
语法 java关键字，内置实现 接口和实现类
锁获取 只能不断重试直到获取锁 更加灵活
锁释放 自动 finally手动释放
中断 不可中断 可中断lockInterruptibly()
公平性 非公平锁 可以控制公平性
条件变量 一个隐式地条件 newCondition()创建多个条件
效率：如果竞争资源不激烈，两者的性能是差不多的，而当竞争资源非常激烈时（即有大量线程同时竞争），此时ReentrantLock的性能要远远优于synchronized。

ReadWriteLock

ReadWriteLock是一个接口，包含两个方法：

  • Lock readLock(); 返回读锁
  • Lock writeLock(); 返回写锁
    ReadWriteLock的目的是在线程安全的基础上提高并发，主要是在并发读的时候不会互相阻塞。读写互斥关系如下：
线程 读 写
读 允许 不允许
写 不允许 不允许
写锁的优先级要比读锁高，有读锁请求进来时，当前写锁释放，写进程优先读线程获得锁。

ReentrantReadWriteLock

ReentrantReadWriteLock是ReadWriteLock的一个实现，可重入。

StampedLock

为了进一步提高并发效率，实现了读写的非互斥，这就是StampedLock在ReadWriteLock上的提升。读写关系改进如下：

线程 读 写
读 允许 允许
写 允许 不允许
StampedLock可以使用乐观锁实现上表关系，ReadWriteLock只能使用悲观锁。StampedLock不可重入。
常用的方法如下：
  • public long readLock() 获得悲观读锁，返回一个时间戳
  • public void unlockRead(long stamp) 解锁悲观读锁
  • public long writeLock() 获得悲观写锁
  • public void unlockWrite(long stamp) 解锁悲观写锁
  • public long tryOptimisticRead() 获得乐观读锁，其实没有加锁，只是获取了一个版本号
  • public boolean validate(long stamp) 验证乐观读，如果从tryOptimisticRead()到这里验证过程中没有悲观写锁被获取过，那么返回true，否则返回false
