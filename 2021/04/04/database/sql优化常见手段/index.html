title:sql优化常见手段

走索引加快查询

  • 根据where条件和order by的字段建立索引。
  • 尽量避免使用!= 或 <> ，in 和not in，会导致数据库引擎放弃索引进行全表扫描。
    • 如果是连续数值，可以用between代替in。
    • 如果是子查询，可以考虑用exists，但是这里还是要具体情况具体分析。一般而言，子查询的表的结果集比较大的话适合用exists，如果子查询的结果集很小，反而用in更快。
  • 尽量避免使用or，会导致数据库引擎放弃索引进行全表扫描。
    • 考虑使用union讲or不同条件的结果集合并。
  • 尽量避免在字段开头模糊查询，会导致数据库引擎放弃索引进行全表扫描。
    • 考虑在字段末尾使用模糊查询
  • where条件中的字段不要使用函数等，会导致数据库引擎放弃索引进行全表扫描。
  • 尽量设置列为非null，空值可以用默认值代替。null可能会导致数据库引擎放弃索引，还有很多其他问题，有待进一步研究。

    减少查询返回数据量

  • 避免使用select *，只返回需要的字段。

    数据库结构优化

    字段类型

  • 设计表的时候，在业务允许的情况下尽量选择占用空间小的列类型。
  • 使用简单的数据类型，在mysql中，int要比varchar处理上简单。
  • 尽量使用not null字段。
  • 尽量少使用text，blob等大字段，非用不可考虑分表。

    范式与反范式

    为了减少表关联，有时候需要进行反范式化设计，通过冗余列，优化查询效率，实际上是使用空间换时间。

    表的垂直拆分

    表的垂直拆分，实际上是将一个有很多列的表拆分成多个表，解决表的宽度问题。基本的拆分原则如下：
  • 把常用字段放到一起，把不常用的字段拆分到一个单独的表。
  • 把大字段拆分到一个单独的表。

    表的水平拆分

    在表中数据量过大时，考虑使用水平拆分。常用的拆分方法是针对某列使用hash，根据hash结果不同到不同的分表。水平拆分一般有两个改造考虑，一是实时查询需要考虑查多表，二是后台数据统计分析需要汇总，但是后台统计时效要求一般较低，所以水平拆分有其价值所在。

    使用数据库内置函数提高查询效率

  • 例如使用mysql的concat()函数会比使用||来进行拼接快，因为concat()函数已经被mysql优化过了。

    数据库配置优化

    Mysql配置优化

  • innodb_buffer_pool_size
    innodb缓冲池大小，推荐配置为总内存的75%。
  • innodb_buffer_pool_instances
    innodb缓冲池数量，默认是1。
  • innodb_log_buffer_size
    innodb日志缓冲区大小
  • innodb_flush_log_at_trx_commit
    控制innodb变更刷新到硬盘的频率，可能的值是0，1，N，默认值为1。0：默认值。事务提交后，将二进制日志从缓冲写入磁盘，但是不进行刷新操作（fsync()），此时只是写入了操作系统缓冲，若操作系统宕机则会丢失部分二进制日志。1：事务提交后，将二进制文件写入磁盘并立即执行刷新操作，相当于是同步写入磁盘，不经过操作系统的缓存。N：每写N次操作系统缓冲就执行一次刷新操作。
  • innodb_read_io_threads/innodb_write_io_threads
    innodb读写线程数，默认为4。
  • innodb_file_per_table
    控制innodb是否每个表使用独立的表空间，默认为off，但是推荐为on。off情况下使用共享表空间，io更容易成为瓶颈；另外使用共享表空间不能收缩，清理数据优化性能有限。
  • innodb_stats_on_metadata
    决定什么情况下刷新innodb表的统计信息。
